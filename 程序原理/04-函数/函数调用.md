# 函数

函数（Function）是什么本文就不用介绍了，能看到这篇文章的同学肯定都有一定的了解。

当一段代码考虑可重用性、可维护性等质量属性的时候就可以将其封装实现为函数，提供给其他代码调用。

本篇文章总结的内容是函数调用的相关知识，为后续计划学习的系统调做铺垫。

## 调用

对于高级语言来说，除了可能要阅读他人为API函数写的注释之外，调用（call）就是顺手的事情。

以C语言为例:

1. 先把函数名写一下

1. 有参数就往括号里写要传的参数
2. 行末补一个分号（；）

然后我们就完成了一次函数调用，朴实无华，光从这个角度我甚至水不出多少字数。

那么在如此简单的表象之下的情况是怎么样的呢？我们用先看一下在函数调用时，机器码做了什么：

```
[root@iZ2zeamih4tp4e8ya0gnukZ 04]# gcc -o func_call.elf func_call.c
[root@iZ2zeamih4tp4e8ya0gnukZ 04]# objdump -S func_call.elf

func_call.elf:     file format elf64-x86-64

......

  40110d:       8b 45 fc                mov    -0x4(%rbp),%eax
```

即使从来没有学习过汇编代码，应该也可以通过C代码的比对看出函数调用发生在main函数的0x40111d这条语句。

为什么我这么认为？因为从objdump的翻译中我们可以看到机器码地址0x401122就是对foo函数符号地址的callq（调用）操作了，那从常识的角度来看之前肯定已经把常数1——参数给准备好了。

可以看到机器码`mov $0x1,%edi`也确实是将0x1写入了寄存器`edi`中的意思。

所以函数调用时，程序会将参数存到寄存器中，然后使用call命令跳转到对应的地址开始执行代码。

可以说这个就是函数调用的流程了，但是还是有一些细节需要深究一下：

1. 函数参数都是写入寄存器`edi`吗？所需的参数如果有多个怎么办？
2. foo函数除了对参数进行相加的逻辑外，开头和结束的、和main函数都有的push、pop、retq机器码是什么意思？

要回答这两个问题，需要了解参数的管理方法与久闻大名的“函数调用栈”的概念。

## 参数存储与调用约定

先给出结论：

1. 当参数个数较少时，调用者会将参数存放在寄存器中传给调用函数
2. 当参数个数较多时，调用者会将参数存储在一个地址中，并且将存储的地址存在寄存器中传给调用函数

千万不能萌生出“函数怎么知道参数是怎么传的”这个问题，因为我们之前的文章已经讲过编译过程。

编译器本身在将代码翻译成机器码时，就已经得知函数的参数信息，因此可以直接决定传参方式。

只要编译的规则是确定的，那么无论是同一个文件中实现的函数还是声明的外部函数，传参的方式都是相同的。

这个规则或者说约定是在应用二进制接口（Application Binary Interface， ABI）进行描述的

>  引用自百度百科：ABI包含了应用程序在这个系统下运行时必须遵守的编程约定。ABI总是包含一系列的系统调用和使用这些系统调用的方法，以及关于程序可以使用的内存地址和使用机器寄存器的规定。

其中与本章节相关的内容有：

- 寄存器的使用

- 参数传递
- 返回值传递

因为我的编译器是x86-64架构，因此我将根据x86-64的ABI文档（20210928）3.2 Function Calling Sequence章节内容的进行描述。

### 寄存器的使用

> ​	This subsection discusses usage of each register. Registers %rbp, %rbx and %r12 through %r15 “belong” to the calling function and the called function is required to preserve their values. In other words, a called function must preserve these registers’ values for its caller. Remaining registers “belong” to the called function.7 If a calling function wants to preserve such a register value across a function call, it must save the value in its local stack frame.

这段话解释了调用者和被调用者在函数调用时对于寄存器使用的约定，它将寄存器分为了两类：

- 被调用者使用后要恢复原本值的寄存器（Callee-saved）

  - 寄存器名称：%rbp，%rbx，%r12，%r13，%r14，%r15
  - 约定：这些寄存器的值在函数调用前后不会变化。如果函数中需要用到，则需要先保存寄存器原本的值，然后在函数返回前还原这些值。

  - 用途：例如上文机器码里的%rbp寄存器，这个寄存器用于维护函数调用栈的地址，被调用者会先使用PUSH将调用者的%rbp值保存，并在返回结束时使用POP还原调用者的%rbp。（这一过程将在“函数调用栈”节进行具体描述）

- 调用者要注意保存原本值的寄存器（Caller-saved）

  - 寄存器名称：除了Callee-saved提到的之外的寄存器（如 %rax，%rcx，%rdx，%rsi，%rdi，%r8–%r11等）
  - 约定：被调用者可以随意修改寄存器存储的值。如果调用者希望这些寄存器的值在函数调用后不变，就得在函数调用前自己把寄存器的值先保存在本地（栈内存），然后函数调用后再给寄存器赋值恢复。

  - 用途：可以用于函数临时参数的传递，例如上文机器码里就将常数存放在了%rdi寄存器中。

### 参数传递

> 对应文档的P22-P25

首先我们不能对参数一概而论，需要对参数做一下分类（Classification）：

- INTEGER：有符号（signed）与无符号（unsigned）的`_Bool`，`char`，`short`，`int`，`long` ， `long long`类型，实际上就是常见的int8_t到int64_t，以及uint8_t到uint64_t。
- SSE：浮点数类型，`_Float16`，`float`，`double`，`_Decimal32`，`_Decimal64`，`__m64`

- ……(还有许多类型，例如SSEUP、X87UP、 COMPLEX_X87、MEMORY等，此处不再介绍)
- 对于结构体（struct）、联合体（union）以及数组（array），将根据定义的成员的类型基于规则计算一个结果类型（resulting class）出来。结果类型的计算规则有多条，这里也不一一例举了。我们需要知道的是结果类型计算出来后，传参的方式就按照结果类型的来。例如有三个参数都是INTEGER类型，而一个参数是MEMORY类型，则结果类型是MEMORY，参数要按MEMORY的传参方式进行传递。

当编译器分析出来所有参数的类型后，将按照参数类型分配寄存器进行传参。

- 如果参数是INTEGER，那就按顺序使用 %rdi、%rsi、%rdx、%rcx、%r8、%r9。

- 如果参数是SSE（浮点数类型），则会按顺序使用%xmm0到%xmm7寄存器。
- ……

例如函数foo需要的参数列表是(int x, int y, int z, double f)，那么我们可以预期到的是foo(1, 2, 3, 0)在x86-64架构下编译出的机器码是mov 0x1, %rdi，mov 0x2, %rsi，mov 0x3, %rdx，mov XXX（浮点数的表示本人记不住），%xmm0。

注：开篇的机器码中的结果是mov 0x1, %edi的原因是%edi是指代%rdi的低32位。对%edi（%rdi）的低32位进行写入时，%rdi的高32位会被自动清零。同类，%esi，%edx都是对应寄存器的低32位。而有时候还能看到%di和%dil，分别是低16位和低8位的意思，对这两个寄存器（%rdi别名）进行写入时，并不会自动把高48位和高56位清零。

INTEGER可以用的寄存器只有6个，如果有7个INTEGER类型要传递的话那该怎么办？由于我们前面已经剧透了会通过栈内存进行传递，因此直接看原文。

> ​	If there are no registers available for any eightbyte of an argument, the whole argument is passed on the stack. If registers have already been assigned for some eightbytes of such an argument, the assignments get reverted.
>
> ​	Once registers are assigned, the arguments passed in memory are pushed on the stack in reversed (right-to-left18) order.







那么在函数调用时，被调用者就可以按照需要读取指定寄存器并拿到指定参数。
