从上一章节链接中，我们已经足够了解经过编译后的对象文件是如何被链接器组装成一个程序，并且也知道了链接器生成的“程序头”就是为了指导操作系统的加载器（Loader）如何来运行这个程序——程序并非是直接拷贝到内存中就完事了：

- 程序的第一条指令（入口函数）的地址是多少？
- 程序初始化的变量都有什么？赋值是多少？
- 程序没初始化的变量都有什么？要开辟多大的全0内存来存放？
- ...

同时也在讲述程序的动态重定位表时知道了程序加载的时候或许还需要一个“动态链接器”的东西来协助。

本章节我们将详细讲述一个程序文件被加载到内存中运行的流程。

## 加载器

当我们讨论说要启动一个程序时，我们已经是在操作系统的范畴下了。而我们其实很能够理解——操作系统本身就是一个运行在机器上的程序。

### 程序，启动

当按下电源键，CPU从固件（如BIOS/UEFI）的入口地址开始执行。固件完成硬件初始化后，会加载存储设备上的引导程序。引导程序将按照预设的条件去硬盘、U盘等存储介质中查找是否存在操作系统并将它加载到内存，并最后移交控制权。在Linux系统中加载的就是内核，当内核初始化完毕后将提供一个交互界面，例如SHELL。

而平时用户便是在SHELL下输入命令启动程序，然后操作系统内核中的加载器就会负责后续的装载流程。

## 读取程序头表

程序的加载是有讲究的,例如程序要读取到内存的哪里?从程序的哪里开始执行?甚至说上一章节我们有讨论到，动态链接是在程序加载时运行，那么该去哪里将这些动态链接的符号加载到内存中供程序访问？

所以除了文件名外，加载器需要更多信息来指导它顺利地装载程序。记得链接器在链接程序时生成的程序头表吗？正是存储在其中的信息在加载程序时发挥了作用。

从回写信息的开头部分我们可以知道，这个程序是一个ELF的可执行程序，程序入口的绝对虚拟地址是0x401030，从文件偏移64字节开始共有11个程序头。

Program Headers部分详细表示了每个程序头的内容：

- Type: 内容的类型，例如INTERP（解释）、LOAD（内容加载）
- Offset: 内容起始在文件中的偏移量
- VirtAddr：加载的虚拟地址
- PhysAddr：加载的物理地址（实际上大部分现代系统的装载器会忽略这个字段，原因见下文）
- FileSiz：内容在文件中的大小
- MemSiz：内容在内存中的大小
- Flags：内容的权限，例如R（读取）、W（写入）、E（执行）
- Align：这部分加载到内存后按多少字节对齐

最下方则是记录了每块内容对应了程序的节头表中的哪些节。

第三个段的类型为Load，.plt节和.text节就在其中。

有了程序表头，加载器就可以将程序按照规定的布局加载到内存中。

所以，程序的内容都已经加载了，那么需要动态链接的部分呢？

## 加载时链接

在上一节中提到，链接器发现符号来自于动态库（.so）时，就会在程序节头表中创建动态重定位表（.rela.dyn节、.rela.plt节）

当上文示例中程序被加载时，加载器发现程序存在`INTERP`的程序头，意味着加载器在加载程序前，需要先启动动态链接器来协助程序完成动态符号的重定向工作。

### 动态链接器

动态链接器是在程序加载时，专门用于执行动态符号重定位流程的程序，程序路径在`INTERP`程序头中已经给出`[Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]`。虽然看起来`ld-linux-x86-64.so.2`这个名字是一个动态库，但实际上它是一个可以运行的静态链接的程序。

加载器会将动态连接器加载到内存中，并从动态链接器的入口处开始执行。

动态链接器程序所做的工作如下：

1. 读取源程序的.dynamic节内容，获取程序依赖的动态库列表
2. 在相关路径下寻找相关动态库并加载到内存中。（相关路径例如系统默认的库路径或用户设置的环境变量等）
3. 首先对动态库进行符号的重定向，因为动态库编译时也是编译为位置无关代码（PIC），需要在加载时更新访问地址
4. 接着根据源程序的.rela.dyn和.rela.plt表中的内容，修正程序中GOT表和PLT表中的条目，使其指向动态库中符号的真实内存地址
5. 调用各个动态库.init节内的逻辑完成初始化
6. 跳转到原本程序的入口函数继续执行

符合之前理解的动态编译器的工作流程。

### 程序整体重定位

加载器基于程序的要求调用动态链接器进行动态链接，动态链接器将程序需要的动态库加载到了虚拟内存空间中完成重定位，那么这些动态库被加载到了哪个内存地址呢？万一那个地址程序要用怎么办？

程序在链接时期待的是自身被加载到0x400000，然后在0x401030的入口函数开始执行，但可能因为各种外部依赖的数据比它更早加载到内存，导致此时虚拟内存中0x400000的位置并不是程序起始位置，加载器再将0x401030当做程序的入口去执行那显然是完全不对的。

因此动态链接器在跳转到原本程序的入口之前，**还需要将程序所有的符号位置基于此时程序自身被加载的内存位置全部做一轮更新**，才可以放心开始执行程序。

对自带fPIE编译选项的程序来说，大部分的更新行为都是更新GOT表项，而对于非fPIE的程序来说，可能要更新的绝对虚拟地址就比较多。

只是有一个问题，为什么程序编译出来一看，内存地址总是使用类似0x400000的地址呢？不会和别的程序冲突吗？假如是真实内存地址，那确实只能有一个人使用，但是得益于操作系统提供的虚拟内存的概念，这个0x400000实际上并不是物理地址而是虚拟内存地址。

## 虚拟内存

假如程序访问的都是物理地址，那么就意味着各个程序需要互相配合，不使用被别的程序计划使用的内存地址，这很明显带来了程序编写、维护、协同工作的巨大挑战。而实际上我们都已经知道了，现代操作系统会将程序加载到固定大小的虚拟内存中，在虚拟内存中程序在32位的设备上最大只能分配4G（2的32次方的寻址空间），而在64位（2的64次方的寻址空间）的设备上，可以说大小已经不是问题。

那么这个虚拟内存地址是怎么工作，是如何和真实的物理地址进行关联的呢？

这个由CPU内部的一个专用硬件模块——内存管理单元（MMU）和操作系统共同配合完成的。

### 内存页

内存页是操作系统对于内存的最小管理单位，以Linux来说，通常情况最小的页大小是4KB，而如果程序希望，操作系统可以支持到GB级别的大内存页。

操作系统只能按页来分配内存的，当程序表示自己需要在虚拟内存中使用1个字节大小的内存，那么操作系统也只能分配1页大小的物理内存给程序，但是通过某些转换让程序得以用虚拟内存地址进行访问。

为什么Linux的页最小是4KB而不是128字节或者直接按每字节映射？这个便涉及到虚拟内存和物理内存是如何对应上的。

### 内存映射

当操作系统为程序分配了一个页后，将通过表映射的方式记录虚拟地址与物理地址的对应关系。又由于每个程序的虚拟内存地址都差不多（例如0x400000），那么操作系统实际上是为每个程序都维护了一张表。这张表称为页表（Page Table）， 表中的表项被称为页表项（Page Table Entry, PTE）。

如果按每字节进行映射，那么等于说1字节虚拟内存需要一个表项对应1字节的物理内存，那页表的大小恐怕都比程序本身大得多。

因此内存映射采用的是块映射的做法，将一块一块的内存进行映射。

假设每个表项只需要存储1个32位的整型，那么大小就是4字节。

如果按128（2的7次方）字节的块大小进行映射，那么一个2的32次方（4G）大小的虚拟内存被描述一共需要2的25次方条表项，那么需要4*2^25字节（128MB）的空间来存储所有表项，显然是过于大了。

如果按4KB（2的12次方）字节的块大小进行映射，那么需要2的20次方条表项，需要4MB来存储表项。这样相对来说就比较能接受一些。

如果按1MB（2的20次方）字节的块大小进行映射，那么需要2的12次方条表项只需要16KB来存储表项。似乎非常合适。

但是块越大，虽然页表占用越小，代价是物理内存分配的多但程序可能压根用不完（内存碎片），造成浪费。因此操作系统在权衡下选择了4KB作为普通一页的大小。

这块用来存储内存映射表项的空间是在操作系统内核为每个程序创建的，因此每运行一个程序，除了程序实际分配的物理内存外，还存在页表存储的消耗。

操作系统是怎么生成并利用这个表项的，我们将在后续进行详细描述，此节只做介绍。页表项中也不仅仅存储了物理地址，还保存了一些重要标志位（读写执行权限位、存在位、脏位等）来支持各种机制的运行。

程序加载时，加载器首先根据程序头，在进程的虚拟地址空间中规划好各个段的布局，并初始化页表。在程序运行过程中需要访问地址0x400000时，MMU会自动计算虚拟地址0x400000对应的虚拟内存页是哪一个，然后通过查表找到对应的物理地址页并完成访问（因为MMU是硬件所以转换效率极高）。

因此对于虚拟内存地址转换为物理地址的工作，实际上是操作系统负责维护表项，MMU负责使用表项做实际访问。而且为了进一步提升转换效率，MMU内部还集成了TLB（转换后备缓冲区）作为页表项的高速缓存，后续也将专门写一个章节来介绍这一关键的优化机制。

### 缺页

所以当程序被加载时，操作系统会建立一张虚拟内存和物理内存之间的页映射表，然后为程序准备好虚拟空间，将程序的内容全部在虚拟内存空间中准备完毕然后运行。

那也就是说每个程序运行时所占用的物理内存大小就是程序头里`Program Headers`声明的大小吗？

其实不一定是这样，因为就像上一章节对动态库函数调用通过GOT表和PLT表配合实现懒加载的机制一样，程序运行的各个环节都在精打细算——如果没用到，那先不加载，假如程序的某些内容当前还未用到，则实际上它并没有被加载到实际的物理内存中。这个是操作系统对程序初始化的一个关键优化措施——惰性加载。

例如程序的代码太长了变量太多了，程序头表里也说.text,.data,.rodata等这些节需要多个页来加载，那么程序运行时入口函数所在的部分肯定要被加载到内存的，但是如果某页所包含的虚拟地址都还没被访问过，那么这个页就可以先不载入内存。而程序是不感知的，当程序使用虚拟内存地址时，MMU在查表过程中发现对应页表项的存在位为0，会立即触发一个硬件异常（缺页中断），CPU随后会跳转到操作系统的中断处理程序中做处理。

缺页中断的处理简单来说做的事情就是：

1. 诊断缺页原因
2. 如果是合法访问且物理页从未分配，则操作系统先在物理内存上分配一块新的物理页
3. 将程序页加载到物理内存中
4. 在映射表中增加该虚拟地址的映射关系，这样程序就能用这个虚拟地址来访问内容
5. 回到程序执行的指令继续执行

上述流程被称为硬缺页（Hard Page Fault），可以看出程序在访问虚拟地址时，对这个地址到底映射了没有是不感知的。

既然说了缺页中断一开始对缺页原因做判断，说明其实还有别的产生缺页的场景，说有合法的场景，那么也就有非法的场景：

- 非法：访问了异常的内存地址，此时会抛出SIGSEGV信号（段错误, Segmentation Fault）
- 合法，但是物理页不存在（存在位为0）：
  - 未被分配——上文中的场景（硬缺页）
  - 被置换了——操作系统会将长期未访问的内存置换到硬盘中暂时释放，等用到的时再从硬盘中加载（另一种硬缺页）
  - 被分配了，只是当前程序没有映射，这种被称为软缺页（Soft Page Fault）

### 动态库共享页

不知道各位是否有听说过，将多个程序共享的代码做成动态库可以减少各个程序在运行时占用的内存大小。

通过上文的介绍来想，因为每个程序的代码变少了，那可能需要的虚拟内存页也变少了，自然需要的物理内存页也变少了。但是动态库的加载也需要物理内存，那好像到头来啥都没省，还更麻烦了？

已知动态库的代码是不会变化的，那么既然每个人访问的都是一样的代码，那把加载了动态库代码的同一页物理页同时映射给每个程序是不是就省下内存了？

实际上，操作系统对于这些不会变化的内存都倾向于采用这种共享页的方式提供给多个程序，例如只读文件也是可以被多个程序共享，此处我们以动态库来举例。

当第一个程序启动时，动态链接器触发了动态库的物理页分配，由于动态库的程序头中表明了这个页是R（只读），那么操作系统就可以将这个物理页标记为只读属性。

当后续程序加载同样的动态库时，操作系统一看这个库已经加载过了并且是只读，没有人将会修订，那么就将这一个物理内存页映射到这些程序中。

注：动态库共享页正是对应了上文中的软缺页场景。

### 写时拷贝

如果程序中程序头属性中说明这一页是W（可写），那么这个页后续所在的虚拟内存区域也会被认为可写。那这些可写的物理页可以共享吗？实际上也可以，当内存页被分配时，最早总是会被设置为只读。当一个程序需要修订其中的内容时，操作系统会先检查这个页对应的虚拟内存区域属性是不是W（可写），如果不行那自然是很大的问题，操作系统将抛出SIGSEGV信号（段错误）。而如果可以写，那么操作系统会重新分配一个全新的内存页，新的内存页处在不同的物理内存上，操作系统将原内存页的数据复制到新的内存页中，然后将程序的内存映射页表项更新为新的内存页。

这样只有这个修订数据的程序使用了区别于那页只读内存页的写副本，原本的只读内存页还是依然能被其他还不做修订的程序共享。

### 多级页表

上文提到如果在32位的系统下，块大小为4KB的页表占用空间为4M。但到了现在64位的系统，如果还想用上述方式记录在一张表里基本是不现实的，因为页表本身需要的空间太大了。

但是话又说回来，为何需要分配一张表来描述全部的地址呢？程序也基本不可能用到那么广的地址范围。如果不记录所有表项而只记录已用到的地址，那么就可以减少页表项，从而减少需要的页表大小。

现代操作系统为了解决“既要记录映射表项，又不能让页表过大的问题”，普遍采用了多级页表——操作系统只为进程实际使用的虚拟地址范围创建中间层页表，未使用的地址空间对应的页表根本不存在，从而极大地节省了内存。

即一开始查询的页表只记录一个大范围，然后通过一级级细分查找的方式找到最终的表项。

以x86-64的四级页表为例，在x86-64中，一个程序的虚拟内存地址最多48位，基于位被划分为了4个阶段的索引：

- `[47:39]`-> 第一级索引，对应的表项指向了第二级页表的地址
- `[38:30]`-> 第二级索引，对应的表项指向了第三级页表的地址
- `[29:21]`-> 第三级索引，对应的表项指向了第四级页表的地址
- `[20:12]`-> 第四级索引，对应的表项指向了目标物理页的地址
- `[11:0]`->目标虚拟内存地址起始位置在这页物理页中的偏移量

当发现某一级的索引对应表项的存在位为0时，说明这类虚拟内存页地址从来没被映射过，下一级的表项压根不可能存在，MMU便不再尝试往下查找，而是触发缺页中断。

这种只记录存在的条目（稀疏存储）的方式，使得页表的总大小与进程实际使用的内存成正比，而非与整个虚拟地址空间成正比。
